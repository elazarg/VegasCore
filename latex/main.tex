\begin{abstract}
VegasCore takes the unusual step of making a dependency graph, not the source
program or an extensive-form tree, the carrier of game-theoretic meaning.
Users write finite games as typed programs.  A checked program is lowered to a
dependency-aware event graph; the graph is interpreted as an asynchronous
probabilistic machine; strategic forms, factored-observation stochastic games
(FOSGs), extensive-form games (EFGs), and backend refinement obligations are
then derived as views of that machine.  The central semantic spine is
\[
  \leanref{Lang.ToCore}
  \longrightarrow
  \leanref{Core.ToEventGraph}
  \longrightarrow
  \leanref{EventGraph.ToMachine}.
\]
On top of this spine, VegasCore exposes native strategy carriers and kernel
games, optional bridges to FOSG and EFG presentations, and refinement
theorems for implementations.  The headline results are frontier commutation
for independent events, preservation of public outcome laws across trace and
EFG presentations, finite mixed-to-behavioural realisation, and Nash transport
to refined backends.  The companion supplement gives the exact Lean
definitions and field lists; this report explains the design and the theorem
surface that users are meant to rely on.
\end{abstract}

\section{Introduction}

Many games are easier to describe as protocols than as normal forms.  A player
commits to a value, a public random sample is drawn, a hidden value is later
revealed, and payoffs are computed from the public transcript.  The mathematical
game is implicit in these operational steps: strategies are constrained by what
players can observe, deviations must be legal at the point where they are made,
and equivalent linearisations of independent events should not produce different
semantics.

VegasCore formalises this protocol-to-game passage in Lean~4
\cite{deMouraUllrich2021}, using the same ecosystem as mathlib
\cite{mathlib2020}.  The user-facing input is a checked Vegas program,
\leanref{WFProgram}.  The semantic owner, however, is not the source syntax
and not a hand-written extensive-form tree.  It is the event graph,
\leanref{EventGraph}: a finite dependency graph whose configurations record
which event nodes have produced which results.  The graph is then interpreted
as a generic observation-aware \leanref{Machine}; strategic objects are
derived from this machine by the FOSG and kernel-game layers.

The immediate benefit of this architecture is that the operational and
strategic stories share the same state space.  A machine trace, a FOSG history,
an EFG presentation, and a backend execution are different presentations of the
same event-graph semantics.  The code does not ask the reader to trust that
several interpreters agree.  Agreement is proved where presentation boundaries
are introduced.

\subsection{A First Example}

Matching Pennies is a useful small example because the source program is
sequential, while the intended game has two simultaneous private choices.  In
the surface language, the program can be read schematically as:
\begin{verbatim}
simultaneous
  yield matcherSecret    matcherPublic    matcher    true
  yield mismatcherSecret mismatcherPublic mismatcher true
ret [
  matcher    := if matcherPublic == mismatcherPublic then  1 else -1,
  mismatcher := if matcherPublic == mismatcherPublic then -1 else  1
]
\end{verbatim}
Each \leanref{yield} line gives a hidden variable, the public variable it will
later reveal into, the acting player, and a guard.  Here the guard is
\leanref{true}, so both players always have both Boolean actions available.
This is the surface \leanref{VegasLang} form; the core \leanref{VegasCore}
lowering replaces the simultaneous block by paired commit/reveal nodes.
The checked Lean example is \leanref{Vegas.Examples.MatchingPennies}; the
surface lowering example is \leanref{Vegas.Examples.VegasLang}.  The
lowered core program is a commit/reveal protocol, but the event graph recovers
the independence that the source tree cannot display:
\[
\begin{array}{ccccc}
\Commit_{\mathit{matcher}} &&&& \Commit_{\mathit{mismatcher}}\\
\downarrow &&&& \downarrow\\
\Reveal_{\mathit{matcher}} &&&& \Reveal_{\mathit{mismatcher}}\\
& \searrow && \swarrow &\\
&& \Ret && .
\end{array}
\]
Initially, the two commits are both frontier nodes.  Executing the matcher
commit first or the mismatcher commit first is only a linearisation choice; it
does not change the event-graph configuration reached after both commits.
The frontier diamond theorem in \S4 is precisely the general statement behind
this example.

\subsection{Terminology}

A probability mass function (PMF) is a finite-support probability
distribution.  A kernel game is the reusable strategic-form carrier from the
GameTheory library: it packages per-player strategy types, an outcome kernel
from profiles to distributions over outcomes, and utilities.  It is close to a
normal-form game, except that outcomes may be probabilistic and the strategy
types need not be hand-written normal-form actions.

A factored-observation stochastic game (FOSG) is a state-based game
presentation in which players act from information states induced by their
observations; the terminology follows Kova\v{r}\'ik et al.'s connection
between FOSGs and EFGs \cite{Kovarik2022}.  An extensive-form game (EFG) is
the usual tree presentation of sequential play.  VegasCore treats both as
presentations of the event-graph machine, not as independent denotations.
Kuhn's theorem is the classical perfect-recall result that mixed strategies
over pure contingent plans can be realised by behavioural strategies
\cite{Kuhn1953}; here it is used in the finite, PMF-valued form stated in
\S7.

\section{Semantic Spine}

The development is organised around semantic ownership rather than around a
single compiler directory:
\[
\begin{array}{c}
\leanref{VegasLang}
  \longrightarrow
\leanref{VegasCore}
  \longrightarrow
\leanref{EventGraph}
  \longrightarrow
\leanref{Machine}.
\end{array}
\]
The compiler-edge modules are:
\begin{center}
\small
\begin{tabular}{@{}lll@{}}
\leanref{Vegas.Lang.ToCore} &
\leanref{Vegas.Core.ToEventGraph} &
\leanref{Vegas.EventGraph.ToMachine}.
\end{tabular}
\end{center}
The strategic and presentation layers then consume the machine:
\[
\begin{array}{ccccc}
\leanref{Machine} & \longrightarrow & \leanref{Vegas.Strategic}
  & \longrightarrow & \leanref{KernelGame},\\
\leanref{Machine} & \longrightarrow & \leanref{Vegas.GameBridge}
  & \longrightarrow & \text{FOSG/EFG views},\\
\text{refined impl. machines} & \longrightarrow & \leanref{Vegas.Backend}
  & \longrightarrow & \leanref{Machine}\ \text{spec}.
\end{array}
\]
The direction is deliberate: each source layer owns its translation to the next
semantic object.

Other folders consume this spine.  \leanref{Vegas.Strategic} contains the
native Vegas strategy carriers and kernel games.  \leanref{Vegas.GameBridge}
contains bridges from the Vegas semantics to external game-form infrastructure,
currently FOSG and EFG.  \leanref{Vegas.Backend} contains operational
refinement obligations for runtimes.  These layers are not interchangeable:
strategic semantics are part of Vegas, while FOSG and EFG are presentations and
bridges.

The main checked contracts are the following.
\begin{enumerate}[nosep]
\item \emph{Lowering correctness.}
  Surface programs lower definitionally to core programs.  The main lowerer is

  The examples record definitional lowerings for the Matching Pennies programs.

\item \emph{Trace-to-public agreement.}
  Projecting blocked traces to public outcomes gives the public pure and PMF

\section{Checked Programs}

A raw \leanref{VegasCore} term is typed syntax.  A \leanref{WFProgram} is the
point where syntax becomes a game description.  It packages the root context,
the program, an initial environment, and the static obligations needed by the
semantic pipeline:
\[
  \leanref{WFCtx},\quad
  \leanref{WF},\quad
  \leanref{NormalizedDists},\quad
  \leanref{Legal}.
\]
The supplement gives the exact structure.  Conceptually, these obligations say:
variables are fresh, visibility scopes are respected, committed values are
eventually revealed, samples are probability distributions, and every commit
site has at least one action satisfying its guard.

The visibility discipline is simple and intentionally strong.  Bindings are
public or hidden to a single owner.  Public computations and payoffs read only
the public erasure of the context, while commit guards are typed over the full
erased context and constrained by \leanref{GuardUsesOnly}.  This separates
expressiveness from information discipline: a guard can be written against the
ambient state, but the well-formedness proof says its truth is determined by the
acting player's view.  For example, a guard may mention a hidden variable such
as a player's own sealed card; \leanref{GuardUsesOnly} is the proof that the
guard's truth cannot change when another player's hidden state changes.

The current surface language, \leanref{VegasLang}, is an elaborated typed layer
over the concrete expression language \leanref{simpleExpr}.  It keeps the core
constructors and adds nullable public yields and simultaneous yield phases.
Lowering is definitional:
\[
  \leanref{VegasLang.lower},\qquad
  \leanref{VegasLang.compileNullable}
  : \leanref{VegasLang} \to \leanref{VegasCore}.
\]
The surface layer therefore improves ergonomics without adding a second
denotation.

\section{Event Graphs}

The event graph is the central object of the development.  It turns a checked
program tree into a finite graph of event nodes, typed storage fields, read
sets, write slices, and dependency edges.  An event-graph configuration is not
a log.  It is an extensional partial assignment from nodes to their result
slices, with proofs that completed nodes are in the graph, closed under
prerequisites, and locally well-shaped.

The graph frontier is computed from this assignment:
\[
  \text{frontier} =
  \{\, n \mid n \text{ is unfinished and every prerequisite of } n
      \text{ is finished}\,\}.
\]
No schedule is stored in the machine state.  This is the main reason the
event graph is a better semantic object than a sequential interpreter: when two
enabled nodes are independent, the state reached after executing both should be
the same regardless of the order in which a presentation chooses to show them.

\begin{theorem}[Frontier preservation]
Let \(C\) be an event-graph configuration.  If \(a\) and \(b\) are distinct
frontier nodes of \(C\), then after executing \(a\) with a legal slice, \(b\)
is still on the frontier of the successor configuration.

In Lean this is
\leanref{EventGraph.Configuration.withResult_mem_frontier_of_ne}.
\end{theorem}

\begin{theorem}[Frontier diamond]
Let \(a\) and \(b\) be distinct frontier nodes of a configuration \(C\).  If
\(s_a\) and \(s_b\) are legal result slices for those nodes, then the two
linearisations agree extensionally:
\[
  (C[a \mapsto s_a])[b \mapsto s_b]
  =
  (C[b \mapsto s_b])[a \mapsto s_a].
\]
In Lean this is \leanref{EventGraph.Configuration.withResult_comm}.
\end{theorem}

These facts are small, but they carry much of the design.  Any strategically
relevant dependency must be represented in the graph.  Once two events share a
frontier, their relative order is presentation data, not semantic data.

\section{Machines and FOSG Views}

\leanref{Machine} is the operational carrier.  A primitive machine event is
either a player action or an internal event; transitions are PMF-valued; states
expose public and player-local observations.  The interpretation
\leanref{EventGraph.toMachine} uses event-graph configurations as states.
Player actions choose a frontier node and a write slice.  Internal events
execute non-player frontier nodes.  Unavailable event-graph actions stutter,
making the step functions total while keeping legality as a separate predicate.

The FOSG layer is a checkpoint view of a machine, not another machine.  A
\leanref{Machine.FOSGView} chooses a round-level action alphabet
\leanref{Act}, active players, legal round actions, and a transition kernel.
For event graphs, a round action can cover every frontier node owned by a
player; the underlying machine still executes primitive events.  The stability
condition \leanref{HasStableFrontierRounds} is the proof obligation that makes
this batching sound.

This distinction matters.  The machine explains execution.  The FOSG view
explains strategic observation and information-state strategies.  The event
graph remains the common object below both.

\section{Strategic Semantics}
\label{sec:strategic}

For a finite checked program \(g\), VegasCore exposes four canonical kernel
games:
\begin{center}
\small
\begin{tabular}{@{}lp{0.46\textwidth}@{}}
\leanref{pureKernelGameAt}\;\(g\) &
  pure strategies, public outcomes,\\
\leanref{pmfBehavioralKernelGameAt}\;\(g\) &
  PMF behavioural strategies, public outcomes,\\
\leanref{pureBlockedTraceKernelGameAt}\;\(g\) &
  pure strategies, blocked machine-trace outcomes,\\
\leanref{pmfBehavioralBlockedTraceKernelGameAt}\;\(g\) &
  PMF behavioural strategies, blocked machine-trace outcomes.
\end{tabular}
\end{center}
The unsuffixed \leanref{pureKernelGame} and
\leanref{pmfBehavioralKernelGame} are the fixed-program public APIs.

The strategy carriers are reachability-pruned and legal by construction.
Illegal deviations are not filtered out after the fact; they are absent from
the carrier over which equilibrium predicates quantify.  Vegas then exposes
the standard solution-concept inventory over these games: Nash equilibrium,
best response, dominance, correlated equilibrium, potential-game predicates,
zero-sum predicates, and welfare quantities.  The representative names are
\leanref{IsNash}, \leanref{IsBestResponse}, and \leanref{socialWelfare};
the full inventory is in supplement \S7.

The blocked-trace games retain the primitive event blocks and endpoint machine
state as outcomes.  They are useful for proving that operational refinements
and public strategic games agree.  The public games are obtained by projecting
blocked traces to public outcomes; the code packages these projections as
\begin{center}
\leanref{KernelGame.Simulation}\qquad
\leanref{KernelGame.Bisimulation}\qquad
\leanref{KernelGame.EUBisimulation}.
\end{center}

\section{Realisation and EFG Presentation}

The main finite realisation theorem is \leanref{kuhn_finite}.  It states that
for any finite checked program, every independent mixed profile over pure
strategies has a PMF behavioural profile with the same public outcome
distribution.  Writing \(\mu\) for the independent mixed profile, \(\beta\) for
its behavioural realisation, and \(K_{\mathit{beh}}\) and
\(K_{\mathit{pure}}\) for the two outcome kernels, the statement has the
shape:
\[
  K_{\mathit{beh}}(\beta)
  =
  \leanref{Math.PMFProduct.pmfPi}(\mu)
    \bind (\lambda \pi, K_{\mathit{pure}}(\pi)).
\]
Here \leanref{Math.PMFProduct.pmfPi} is the independent product PMF over the
players' mixed strategies.
The supplement records the exact Lean types of these games and their outcome
kernel fields.
The project-facing property \leanref{KuhnPMF} and theorem
\leanref{kuhnPMF_finite} record the same fact as a named protocol-level
claim.

VegasCore also exposes an EFG bridge, but the bridge is intentionally not the
semantic owner.  \leanref{eventGraphEFGAt} applies GameTheory's canonical
FOSG-to-augmented-EFG construction to the bounded event-graph FOSG.  Theorem
\begin{center}
\small\leanref{eventGraphEFGAt_outcomeKernel_map_publicOutcome}
\end{center}
proves that the translated EFG behavioural profile induces the same public
outcome law as the Vegas behavioural kernel game.  The EFG is therefore a
checked presentation of the event-graph semantics, not an alternate
interpreter.

\section{Backend Refinement}

The backend layer is for implementations that have more machinery than the
source language.  A contract runtime, for example, is not an event graph: it
may have transactions, mempool order, scheduler choices, batching, network
state, or runtime bookkeeping.  Such implementations are represented as
machines and related to the canonical event-graph machine by refinement.

\leanref{WeakStepRefinement} is a one-step simulation relation.  Implementation
events either project to specification events or are internal stuttering steps
under the projected specification state.  \leanref{StochasticStepRefinement}
is stronger: it preserves the projected next-state PMF exactly.  The latter is
the relation used for probability-preserving backend results.

The backend kernel-game layer packages a refinement plus block-law lifts as
EU-preserving morphisms.  The blocked-trace pullback and behavioural pullback
theorems,
\begin{center}
\scriptsize
\begin{tabular}{@{}l@{}}
\leanref{backendBlockedTraceKernelGameAt_isNash_pullback}\\
\leanref{backendPMFBehavioralBlockedTraceKernelGameAt_isNash_pullback},
\end{tabular}
\end{center}
say that Nash equilibria established for the canonical spec game transport to
the backend blocked-trace games under the supplied refinement data.  This is
the intended verification workflow: reason about the canonical Vegas semantics,
then prove that a runtime implements it.

\section{Worked Example Revisited}

The Matching Pennies example from the introduction reaches all three public
semantic views.  The checked program \leanref{Vegas.Examples.MatchingPennies.game}
is a \leanref{WFProgram}; its generated public games are
\leanref{pureKernelGame} and \leanref{pmfBehavioralKernelGame}.  The theorem
\leanref{payoffs_zero_sum} checks the intended payoff invariant directly on
the public payoff expressions:
\[
  u_{\mathit{matcher}}(h,t) + u_{\mathit{mismatcher}}(h,t) = 0.
\]

The event graph supplies the concurrency story: both commits are initially
enabled, each reveal depends only on the corresponding commit, and the payoff
node depends on both reveals.  The FOSG view supplies the strategic story:
each player's strategy is indexed by the observations available at its
decision point.  The EFG bridge supplies the conventional tree story:
\leanref{eventGraphEFGAt} serialises each FOSG round into decision nodes and
chance, and the public outcome law of the translated EFG is the same law as
the Vegas behavioural kernel game.

Thus the example can be read in three equivalent ways.  As a program, it is a
commit/reveal implementation of simultaneous play.  As an event graph, it has
two independent initial actions.  As a kernel game, it is the familiar
zero-sum Matching Pennies game, with mixed pure strategies and behavioural
strategies related by \leanref{kuhn_finite}.

\section{Related Work and Scope}

VegasCore sits between programming-language semantics and formal game theory.
Normal-form games are compact but erase protocol order and observations.
Extensive-form games expose order, but force independent events into an
arbitrary tree order.  FOSGs are closer to the state-based operational view
\cite{Kovarik2022}, and the GameTheory library in this repository supplies
the reusable FOSG, EFG, and kernel-game infrastructure.  VegasCore's
contribution is the checked compiler path from visibility-aware programs to
event graphs and then to those game-theoretic presentations.

The scope is intentionally finite.  Player sets, relevant value domains,
program syntax, event graphs, and strategy carriers are finite in the theorem
surface described here.  Distributions are finite-support PMFs; continuous
types and measure-theoretic kernels are out of scope.  The backend layer states
refinement obligations and proves transport theorems, but it does not claim
that every concrete runtime model has already been supplied.  Mechanism-design
libraries can be connected through the generated kernel games, but they are not
part of the core compiler semantics.

\section{Conclusion}

The Matching Pennies example is the small picture: a sequential program, an
event graph with independent initial commits, and a familiar zero-sum strategic
game all denote the same object.  The broader design is to make that pattern
the default: source programs define event graphs, event graphs define machines,
and every game-theoretic or backend view proves how it projects back.

Natural next steps are richer surface notation, concrete backend models,
larger mechanism examples, and eventually infinite-horizon or measure-theoretic
extensions.  The current report is deliberately narrower: finite programs,
finite-support probability, and checked presentation boundaries.
