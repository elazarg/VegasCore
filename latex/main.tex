\begin{abstract}
VegasCore is a small typed language for describing finite games as
protocols: public samples, sealed player commitments, public reveals,
and terminal payoff expressions.  The central semantic object is not
the source syntax and not a hand-written extensive-form tree, but a
finite visibility-aware \emph{event graph}.  Its nodes are protocol
events; its fields are typed values; and its dependency relation records
exactly the information that must be available before an event can run
or before a player can choose a commitment.

The current construction separates three levels.  A \emph{graph
program} contains exactly the static data needed to compile to a graph:
unique initial names, fresh binders, and normalized samples.  A
\emph{checked program} adds the game-facing obligations: reveal
completeness and commit-guard liveness.  The compiled graph is
source-independent and graph-native: administrative surface
\(\Let\)-bindings are erased before core compilation, graph fields are
computed from initial rows and event rows, and prerequisites are derived
canonically rather than stored as independent semantic data.

The primitive graph machine is schedule-agnostic: schedules are external
event laws over currently available primitive events.  Game-theoretic
semantics are built from a canonical frontier round view, which hides
irrelevant primitive linearization and exposes player choices at
checkpoints.  Native pure, behavioral, and product mixed-pure frontier
kernel games are the Vegas-facing strategic denotations.  Bounded FOSG
and EFG objects are derived presentations with payoff-adequacy theorems,
not independent owners of the semantics.

The mechanized theorem surface proves graph well-formedness of
compilation, checkpoint progress from checked guard liveness, terminal
source-payoff adequacy, native/FOSG/EFG utility-distribution adequacy,
finite Kuhn strategic equivalence for frontier strategies and their
one-player deviations, and runtime trace adequacy for stochastic
refinements.  The refinement layer is runtime-general: an EVM backend is an intended
consumer, but the language, event graph, machine, and strategic
interfaces do not mention any blockchain-specific runtime.
\end{abstract}
