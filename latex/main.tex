\begin{abstract}
VegasCore is a Lean~4 mechanization of finite games written as programs.
A Vegas program combines deterministic computation, finite-support sampling,
strategic commitments, and explicit reveals of private values. The core
invariant is that information is represented in the program context: a binding
is public, or hidden and owned by one player. Strategy spaces are indexed by
the player's view of that context, so legal strategies talk only about
information that the player can observe.

The development exposes several equivalent views of a checked Vegas program.
The canonical outcome evaluator is the denotational distribution
\(\outcomeDist\). Trace semantics decomposes that distribution into complete
execution paths. Raw small-step semantics gives a labelled operational
presentation and is proved to agree with the same outcome evaluator. Strategic
and protocol-level APIs package the outcome semantics as game-theoretic
solution concepts. The FOSG bridge gives the canonical sequential
game-theoretic denotation and supplies the finite mixed-to-behavioral
realization theorem used by the public Kuhn-style results.
\end{abstract}

\section{Overview}\label{sec:overview}
%% =====================================================================

VegasCore connects a programming-language presentation of games to standard
game-theoretic objects. The programming side provides syntax, typed
environments, visibility annotations, and executable outcome distributions.
The game-theoretic side provides strategy profiles, kernel games, equilibrium
concepts, and finite sequential-game theorems.

The central user-facing object is a checked program:
\[
  g : \WFP.
\]
A checked program packages a Vegas program, an initial environment, and the
facts required to treat the program as a finite game. From \(g\), the library
derives legal strategy spaces, strategic-form games, protocol-native
definitions, trace and small-step semantics, and a sequential FOSG denotation.
These are not competing meanings of the program. They are different interfaces
to the same outcome semantics, with bridge theorems recording the agreement.

\section{The Program Model}\label{sec:model}
%% =====================================================================

\subsection{Contexts and Visibility}

The core syntax is indexed by a visibility context:
\[
  \VCtx : \mathrm{List}\;(\mathrm{VarId} \times \mathrm{visibility}).
\]
A binding is either public, written \(\pub\), or hidden and owned by a player,
written \(\hid(i)\). This annotation is the main information-flow device in
VegasCore. A player's view of the context keeps public variables and that
player's hidden variables; other hidden variables are not available.

The same idea appears at the environment level. A full environment contains
all realized values. A player view environment is a projection of the full
environment to the variables visible to that player. Commit strategies consume
view environments, not full environments, which is what prevents a strategy
from depending on hidden information owned by someone else.

\subsection{Core Constructs}

The core language has five constructs:
\begin{itemize}[nosep]
\item \(\Ret\): terminate with payoff expressions;
\item \(\Let\): compute a public deterministic binding;
\item \(\Sample\): sample a public value from a finite distribution;
\item \(\Commit\): let a player choose a hidden value satisfying a guard;
\item \(\Reveal\): publish a previously hidden value.
\end{itemize}

\(\Let\) and \(\Reveal\) are deterministic administrative steps. \(\Sample\)
is nature's probabilistic step. \(\Commit\) is the strategic step. The final
payoff expressions are evaluated in the public environment at \(\Ret\).

\subsection{Checked Programs}

The public boundary is \leanref{WFProgram}. A checked program contains:
\begin{itemize}[nosep]
\item the root context, program, and initial environment;
\item freshness and scoped visibility facts;
\item distribution normalization for all samples;
\item commit legality, meaning every guard admits at least one legal action.
\end{itemize}

This boundary is deliberately strong. Once a program has crossed it, downstream
game APIs can quantify over legal strategy spaces without restating basic
program well-formedness at every theorem. The proof obligations also prevent
deadlock in the sequential denotation: every nonterminal checked state has a
legal continuation.

\section{Outcome Semantics}\label{sec:outcome}
%% =====================================================================

\subsection{Finite Distributions}

The raw evaluator uses \(\FDist{A}\), a finite weighted distribution over
values of type \(A\). Its weights live in \(\mathbb{Q}_{\ge 0}\). Normalization
is proved separately and is available after a program is checked. PMF-based
interfaces are used where the surrounding game-theory library expects
probability mass functions.

\subsection{Canonical Evaluator}

The canonical raw outcome evaluator is:
\[
  \outcomeDist\; \sigma\; p\; env : \FDist{\Outcome}.
\]
Here \(\sigma\) is an omniscient operational profile. The evaluator recurses
through the program:
\begin{itemize}[nosep]
\item \(\Ret\) evaluates payoff expressions;
\item \(\Let\) extends the environment with the deterministic value;
\item \(\Sample\) binds over the sample distribution;
\item \(\Commit\) binds over the profile's commit distribution;
\item \(\Reveal\) copies a hidden value into a public binding.
\end{itemize}

The evaluator is the reference semantics for raw execution. Other raw
presentations either compute a projection of the same choices or are stated as
agreement theorems against this evaluator.

\subsection{Trace Semantics}

\leanref{Vegas.TraceSemantics} defines complete execution traces. A trace
records the realized values at every \(\Sample\) and \(\Commit\) site.
Deterministic \(\Let\) and \(\Reveal\) nodes are present structurally but do
not add probabilistic choices.

The trace layer provides:
\begin{itemize}[nosep]
\item \leanref{traceOutcome}, the outcome reached by a complete trace;
\item \leanref{traceWeight}, the product of sample and commit weights;
\item \leanref{traceDist}, the distribution over complete traces;
\item reachability predicates such as \leanref{CanReach} and \leanref{Reach};
\item commutation results for independent commits and reveals.
\end{itemize}

The key relationship is that summing trace weights by outcome recovers
\(\outcomeDist\). This makes trace semantics the pathwise explanation of the
canonical evaluator.

\subsection{Raw Small-Step Semantics}

\leanref{Vegas.SmallStep} defines a labelled one-step relation over neutral
runtime worlds:
\[
  \mathrm{Step}\; \sigma\; w\; d.
\]
A world is a program suffix paired with an environment. A one-step transition
returns a finite distribution over labelled target worlds. Labels are
\(\tau\) for deterministic administrative steps, sample labels for nature
values, and commit labels for strategic values.

There are two small-step notions:
\begin{itemize}[nosep]
\item \leanref{Step} records the outgoing one-step distribution;
\item \leanref{Steps} records qualitative multi-step reachability through
      positive-mass supported edges.
\end{itemize}

\leanref{Steps} intentionally consumes \leanref{StepSupport}, not bare
\leanref{Step}. A multi-step edge therefore represents a target with positive
one-step mass. The relation itself is qualitative; the probabilistic mass of
complete labelled runs is carried by \leanref{labelDist}.

The raw small-step evaluator is:
\[
  \mathrm{runSmallStep}\; \sigma\; w
    \coloneqq \outcomeDist\; \sigma\; w.prog\; w.env.
\]
This definition makes \(\outcomeDist\) canonical and avoids a second recursive
outcome evaluator. The small-step content is supplied by the theorem
\leanref{step_bind_runSmallStep}, which states that one supported operational
step followed by \leanref{runSmallStep} has the same outcome value as the
source world.

\subsection{Trace and Small-Step Agreement}

The small-step labels are a projection of traces:
\[
  \mathrm{traceLabels}\;p\;t : \mathrm{List}\;(\mathrm{Label}).
\]
The theorem \leanref{labelDist_eq_traceDist_map_traceLabels} states that the
complete-run label distribution is exactly \(\mathrm{traceDist}\) mapped
through this projection. Support lemmas then connect positive-mass label runs
to positive-weight traces, and admissibility turns those traces into legal
execution witnesses.

Together, the raw semantic picture is:
\[
  \mathrm{labelDist}
    = \mathrm{map}\;\mathrm{traceLabels}\;\mathrm{traceDist},
  \qquad
  \mathrm{runSmallStep}
    = \outcomeDist.
\]
The first equation relates the observable operational labels to traces. The
second says that outcome evaluation remains the canonical denotation.

\section{Strategic and Protocol Views}\label{sec:strategic}
%% =====================================================================

\subsection{Legal Strategy Spaces}

Commit guards constrain the values a player may choose. VegasCore reflects
that constraint in the strategy types:
\begin{itemize}[nosep]
\item \leanref{LegalProgramBehavioralStrategy};
\item \leanref{LegalProgramBehavioralProfile};
\item \leanref{LegalProgramPureStrategy};
\item \leanref{LegalProgramPureProfile};
\item PMF variants for probabilistic behavioral strategies.
\end{itemize}

The public APIs therefore quantify over guard-legal strategies. This keeps
protocol legality out of theorem side conditions and makes illegal deviations
unrepresentable in the usual equilibrium statements.

\subsection{Strategic-Form Kernel Games}

The behavioral strategic-form wrapper is \leanref{toKernelGame}. The pure
strategic-form wrapper is \leanref{toStrategicKernelGame}. Both are
\leanref{GameTheory.KernelGame}s whose outcome kernels are induced by the
Vegas outcome semantics.

The strategic wrapper files expose Vegas-facing names for standard solution
concepts: Nash equilibrium, strict Nash equilibrium, best response, dominance,
correlated equilibrium, welfare, approximate equilibrium, potential games, and
zero-sum or constant-sum properties. Corollary files transport generic
\leanref{GameTheory} theorems through these wrappers.

\subsection{Protocol-Native Statements}

\leanref{Vegas.ProtocolSemantics} states strategic concepts directly over the
program's outcome semantics. This is useful when the desired statement should
refer to the protocol rather than to a strategic-form wrapper.

The correspondence theorems say that the two presentations agree:
\begin{itemize}[nosep]
\item \leanref{ProtocolNash} agrees with \leanref{IsNash};
\item \leanref{ProtocolBestResponse} agrees with \leanref{IsBestResponse};
\item \leanref{ProtocolDominant} agrees with \leanref{IsDominant};
\item \leanref{ProtocolStrictNash} agrees with \leanref{IsStrictNash}.
\end{itemize}

Thus the protocol-facing definitions are not a separate game model. They are
the same outcome semantics presented without first naming the kernel-game
wrapper.

\section{Sequential FOSG Denotation}\label{sec:fosg}
%% =====================================================================

\subsection{Role of FOSG}

The FOSG bridge is the canonical sequential game-theoretic view of a checked
Vegas program. It treats the program as a finite horizon game with
state-dependent action availability and observation-indexed behavioral
strategies.

This layer is game-theoretic rather than foundational for the raw runtime.
The neutral runtime lives in \leanref{Vegas.Config}. The FOSG layer imports
that runtime, adds active-player and action-availability structure, constructs
a finite observed game, and proves that running the observed game preserves
the Vegas outcome distribution.

\subsection{Runtime, Actions, and Cursors}

The FOSG code is split into layers:
\begin{itemize}[nosep]
\item \leanref{Vegas.FOSG.Runtime}: active players, broad action availability,
      terminality aliases, and joint-action legality over neutral worlds;
\item \leanref{Vegas.FOSG.Action}: commit cursors and finite program-local
      actions;
\item \leanref{Vegas.FOSG.Basic}: suffix and cursor machinery, checked
      worlds, transitions, and preservation facts;
\item \leanref{Vegas.FOSG.Observation}: the observed FOSG target;
\item \leanref{Vegas.FOSG.Observed.*}: behavioral, pure, collapse,
      refinement, and kernel proofs;
\item \leanref{Vegas.FOSG.SmallStep}: checked PMF small-step aliases and
      bridge lemmas.
\end{itemize}

The broad action type is useful for local semantic statements: a player action
is a typed value. The executable FOSG target uses \leanref{ProgramAction}
instead. A program action is a commit cursor paired with a value of that
commit's type. This avoids requiring the entire language universe of types to
be finite; only the values that occur at program-local commit sites matter.

\subsection{CheckedWorld and CursorCheckedWorld}

There are two checked state representations because they serve different
proof obligations.

\leanref{CheckedWorld} is the proof-friendly semantic state. It stores the
current program suffix, environment, and local obligations needed by the
checked transition: context well-formedness, freshness, scoped visibility,
normalization, and legality. This is convenient for recursive semantic
arguments and for relating checked PMF steps to \(\outcomeDist\).

\leanref{CursorCheckedWorld} is the finite executable carrier for the observed
FOSG. It keys states by a \leanref{ProgramCursor} into the fixed root program
and an environment at that cursor. The cursor identifies the current
continuation inside the original program, which is needed for a fixed global
strategy profile to be projected to the current suffix. The carrier is finite
under finite value types because cursors are drawn from one finite program and
environments range over finite value sets.

The two representations are connected by projections such as
\leanref{ofCursorChecked}. FOSG execution uses the cursor-keyed carrier;
semantic comparison often projects back to checked worlds.

\subsection{Observation-Preserving FOSG}

\leanref{toFOSG} exposes the public sequential denotation. Internally,
\leanref{observedProgramFOSG} uses cursor worlds, program-local actions, and
private observations consisting of the player's current program cursor and
view environment. The public observation channel is trivial because public
bindings already appear in every player's private view.

The bridge proves:
\begin{itemize}[nosep]
\item legal observability for the compiled FOSG;
\item bounded horizon by \leanref{syntaxSteps};
\item finite world and action instances under \leanref{FiniteValuation};
\item transition agreement after forgetting cursor keys;
\item outcome preservation after mapping terminal histories to Vegas outcomes.
\end{itemize}

\subsection{Checked Small-Step Bridge}

\leanref{Vegas.FOSG.SmallStep} exposes the checked PMF one-step kernel as
\leanref{stepPMF}. This is the legal, view-indexed counterpart of the raw
\(\FDist\)-valued small-step layer. Its continuation value is
\leanref{outcomeValuePMF}.

The theorem \leanref{stepPMF_bind_outcomeValuePMF} is the checked analogue of
raw one-step value preservation. The theorem
\leanref{legalActionLaw_bind_transition_eq_stepPMF} states that the FOSG legal
action law followed by the FOSG transition collapses to this checked step after
forgetting cursor keys. Finally,
\leanref{mappedRunDist_eq_toKernelGamePMF} connects the observed FOSG run
distribution to the PMF strategic game API.

\subsection{Kuhn-Style Realization}

The public finite mixed-to-behavioral realization theorem is:
\[
  \leanref{kuhn_mixedPure_realizedByBehavioralPMF_finite}.
\]
It says that, for a finite checked Vegas program, every independent mixed
profile over legal pure strategies has a legal PMF behavioral profile with the
same distribution over payoff outcomes.

The proof route is:
\[
\begin{array}{c}
\text{mixed legal pure Vegas profile} \\
\downarrow \\
\text{observed finite FOSG} \\
\downarrow \\
\text{FOSG behavioral witness preserving terminal-history outcomes} \\
\downarrow \\
\text{collapsed legal Vegas behavioral profile preserving } \Outcome.
\end{array}
\]
The preservation statement is distributional. Expected-utility equalities are
corollaries obtained by mapping utilities over the equal outcome distribution.

\section{Design Principles}\label{sec:principles}
%% =====================================================================

\paragraph{Keep the checked boundary explicit.}
Raw syntax is useful for constructing examples and proving structural lemmas,
but public game-theoretic APIs should consume \leanref{WFProgram}. This keeps
freshness, scoped visibility, normalization, and non-deadlock assumptions in
one place.

\paragraph{Make illegal strategies unrepresentable.}
The legal strategy spaces are subtypes whose choices satisfy commit guards.
Equilibrium and dominance statements therefore quantify over legal deviations
by construction.

\paragraph{Keep \(\outcomeDist\) canonical.}
Trace semantics, labelled small-step semantics, strategic wrappers, and the
FOSG bridge all point back to the same payoff-outcome distribution. This avoids
having multiple recursive outcome evaluators that later need to be declared
equivalent.

\paragraph{Treat FOSG as a sequential game view.}
FOSG is the right abstraction for finite sequential-game reasoning: histories,
observations, legal actions, bounded horizon, and Kuhn-style realization. It
should depend on the Vegas runtime and checked semantics, not define them.

\paragraph{Separate semantic state from finite execution keys.}
\leanref{CheckedWorld} is convenient for semantic preservation. 
\leanref{CursorCheckedWorld} is convenient for finite observed-game execution.
The projection between them keeps both proof styles available without forcing
one representation to serve every purpose.

\section{Conclusion}\label{sec:conclusion}
%% =====================================================================

VegasCore gives checked Vegas programs a single canonical payoff-outcome
semantics and several proved-compatible interfaces. Trace and small-step
semantics explain execution structure. Strategic and protocol APIs expose the
same semantics as solution concepts. The FOSG bridge supplies the sequential
game-theoretic denotation and the finite mixed-to-behavioral realization
result, then collapses the witness back to a native guard-legal Vegas
behavioral profile.
