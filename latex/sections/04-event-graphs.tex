\section{Event graphs}
\label{sec:eventgraphs}

The event graph is the dependency-structure IR produced by core
compilation---the mechanism through which meaning is factored, not a
semantic object competing with the program.  It is small on purpose.  It does not store source names, source occurrences,
rank metadata, custom patch legality predicates, or action kernels
parallel to node semantics.  Those are either source-layer data or
derived from the node row.  The graph is therefore a typed dependency
graph, not a syntax tree: it records what must happen and what
information each event may inspect, not how the programmer happened to
spell the source program.

\subsection{Graph data}
\label{sec:eg:data}

An event graph is a pair of lists:
\[
  \EG = (\mathsf{initialFields},\mathsf{nodes}).
\]
An initial field row contains a type, an optional owner, and an initial
value.  An event node row contains the type and owner of its output
field together with a semantic payload:
\[
  \mathsf{NodeSem} ::= \mathsf{sample}(D)
    \mid \mathsf{commit}(p,R)
    \mid \mathsf{reveal}(f).
\]
Event-graph node payloads use lowercase sans-serif names to distinguish
them from the source-language constructors \(\Sample\), \(\Commit\),
and \(\Reveal\).
Initial field ids are \(0,\ldots,|\mathsf{initialFields}|-1\).  The
output field of node \(n\) is
\[
  \mathsf{nodeTarget}(n) = |\mathsf{initialFields}| + n.
\]
Thus event-field writers are not stored as a second map: the writer of
an event field is determined by arithmetic on the graph row.

The node payloads use typed field references.  Payload evaluation is not
given the whole store.  Instead it receives a \(\mathsf{ReadEnv}\) for
its declared typed read footprint.  This is the boundary that makes the
read set semantic: graph-local code can only read fields that appear in
its footprint.  If a runtime or presentation later claims that two events
are independent, that claim is checked against these footprints.

\subsection{Reads and well-formedness}
\label{sec:eg:wf}

The read footprint controls both scheduling and observation:
\begin{itemize}\itemsep1pt
\item a sample reads the public fields declared by its distribution;
\item a commit choice footprint contains the fields visible to the
  committing player at the commit site;
\item a reveal reads the sealed source field being opened.
\end{itemize}

The commit footprint is the player's visible context at the commit site,
not the guard expression's syntactic dependencies.  A guard may ignore
some of that environment; scheduling and player observation still use the
larger choice footprint, because strategies may condition on the whole
visible commit-site view.

A sample is the dual case.  Its scheduling and independence footprint is the
declared dependency set of its distribution, $\mathrm{deps}(D)\cap\pub$, which
is typically a \emph{small} subset of the public store.  The small-step
\textsc{Sample} rule (\Cref{fig:vegas-sos}) writes $\sem{D}$ over the whole
public environment $\rho^{\pub}$ only for notational economy: by the
dependency-soundness law (\Cref{sec:prelim:contexts}) the result depends only
on $\mathrm{deps}(D)$, so the two readings agree on values while the graph uses
the narrower footprint for scheduling.  This is what lets two samples with
disjoint declared footprints commute, even though both nominally ``read public
state.''

Graph well-formedness is the conjunction of the following facts: for
each node, every read field must already be available; each read field
is either initial or written by an earlier node; internal events with
public output read only public fields; commit choice footprints are
public or owned by the committing player; and a reveal opens a sealed
field into a public output field of the same type.

Graph reads are not raw SSA ancestry or a full program-dependence graph
\citep{ferrante1987program}.  They are the information dependencies that
can affect scheduling and player observation.  Pure administrative
source lets are erased before this layer, so they do not create
observable graph nodes.

\subsection{Canonical prerequisites}
\label{sec:eg:prereqs}

Prerequisites are not stored in graph data.  They are derived
canonically:
\[
\begin{aligned}
  q \in \prereq(n)
  \quad\Longleftrightarrow\quad
  &q < n \;\wedge\;
    \bigl(\mathsf{nodeTarget}(q) \in \readsOf(n)\\
  &\vee\;
    (\mathsf{kind}(n)=\mathsf{reveal}
     \wedge \mathsf{kind}(q)=\mathsf{commit})\bigr).
\end{aligned}
\]
The first disjunct is the ordinary information dependency: an event
that reads a field waits for the event that wrote the field.  The
second disjunct is the commit/reveal barrier.  It makes a reveal wait
for prior commits even when the reveal does not read their payloads.
This is what enforces the simultaneous-commit discipline in protocols
such as sealed bidding and Matching Pennies.

Making prerequisites canonical removes a source of unsoundness.  The
compiler cannot accidentally store one dependency relation while the node
semantics imply another one.  Given the list of nodes and their read
footprints, the prerequisite relation is determined.

The negative cases are as important.  Commit/commit pairs are
independent unless the later commit choice footprint contains a field written
by the earlier one.  Reveal/reveal pairs are independent at this abstraction
level unless one reveal reads a field written by the other, which the
core language does not generate.  Public samples do not wait for sealed
commitments.

\subsection{Configurations and availability}
\label{sec:eg:configurations}

A configuration stores a finite set of completed nodes and a typed field
store:
\[
  C = (\mathsf{done},\mathsf{store}).
\]
The initial configuration has no completed nodes and contains exactly the
initial field values.  Completing a node records the node as done and
writes its canonical output field.  There is no operation that completes
a node while writing an arbitrary field.

A \emph{raw} configuration is this graph data: a completed-node set and
a store.  A \emph{reachable} configuration is a raw configuration paired
with evidence that it arises from the initial configuration by legal
primitive steps.

A node is ready when it is not done and all canonical prerequisites are
done.  A commit action is available when the node is ready, owned by the
acting player, the action value has the guard type, the guard's choice
environment can be built from the active store, and the guard evaluates
to true.  An internal event is available when a ready sample or reveal can
evaluate from the active store.

\subsection{Schedule independence}
\label{sec:eg:diamond}

The graph layer proves local schedule-independence facts.  If two
distinct available events are ready at the same configuration, completing
one preserves availability of the other, because ready nodes do not read
each other's output fields.  Supported results for the two events, i.e.\
events or result branches in the support of the relevant finite
distribution, can be swapped to reach the same raw graph configuration.

These theorems do not assert that every real runtime hides every
implementation trace.  They say that the graph state and graph
observations are functions of the completed-node downset and field
values, not of an irrelevant ordering witness.  Strategic presentations
can therefore quotient or checkpoint primitive order without changing
the graph-level state they observe.

This is the exact point where implementation work will later enter.  If
a backend exposes timing, gas, mempool ordering, or other details, that
extra information is not silently ignored by the graph semantics.  It
must either be absent from the implementation observations being modeled
or be accounted for in a runtime-level analysis/refinement assumption.

\begin{theorem}[Frontier diamond]
\label{thm:diamond}
For two distinct primitive events available at the same reachable
configuration and supported by the canonical event law, there exist two
length-two batches, one in each order, whose final raw graph
configurations agree.  Consequently the public and player graph
observations at the two checkpoints are equal.
\end{theorem}

\begin{proof}[Mechanized proof]
See \TT{Vegas.Theorems.Execution}; the named result is the
compiled-graph adjacent-schedule diamond.
\end{proof}

\Cref{thm:diamond} is local: it commutes two adjacent events.  Because a
configuration is the completed-node set together with the canonically-written
store (\Cref{sec:eg:configurations}), and that data is extensional --- no
ordering witness is recorded --- adjacent commutation lifts to a global
statement: any two complete schedules of the same graph reach the same terminal
configuration, so the terminal state is a function of the graph alone.  This
full-schedule confluence, not merely the pairwise diamond, is what the frontier
round model relies on.

\begin{theorem}[Commit/reveal barrier]
\label{thm:barrier}
If a reveal node is ready, every earlier commit node is already complete.
Equivalently, a prior commit action is not available at a checkpoint where
the reveal is ready.
\end{theorem}

\begin{proof}[Mechanized proof]
See \TT{Vegas.EventGraph.Basic} and \TT{Vegas.Theorems.Visibility};
the key graph lemma is \TT{prior\_commit\_mem\_prereqs\_of\_reveal}.
\end{proof}

\subsection{Compiler adequacy at the graph boundary}
\label{sec:eg:adequacy}

Compilation produces a \(\mathsf{CompiledProgram}\) containing:
\begin{itemize}\itemsep1pt
\item a well-formed graph;
\item payoff projections as graph-local \(\mathsf{EventPayoff}\)
  entries;
\item a proof that payoff reads are public and available by graph
  termination.
\end{itemize}
For terminal reachable primitive-machine states, payoff projection is
total and agrees with the source payoff expressions reconstructed from
the terminal compiled store.
